# 多端产品与权限规划

> 面向 Web、Android、iOS 与管理端的下一阶段产品规划
>
> 目标：把当前已经可用的 API 管理能力，扩展成一个统一的、多端可用的生成应用。

---

## 1. 背景

当前系统已经具备以下基础能力：

- 工作流导入、解析、参数配置
- 任务提交、执行、排队、状态查询
- 模型依赖检测与下载
- 用户登录、注册、积分/点数控制
- 管理端基础能力可用

下一阶段要做的，不再是单点功能补丁，而是把这些能力整理成一套统一的产品体系，供：

- Web 主站
- Android App
- iOS App
- 管理端

共同使用。

---

## 2. 产品目标

产品最终形态建议接近“即梦 / 可灵”这类体验：

- 用户可以登录、注册、查看剩余点数
- 用户可以浏览可用工作流
- 用户可以按工作流提交任务
- 用户可以查看任务排队、执行进度和历史结果
- 用户可以删除自己生成的资源
- 管理员可以控制哪些工作流对哪些用户开放
- 管理员可以按用户、用户级别、用户组进行授权

---

## 3. 核心原则

### 3.1 一套后端契约，多端复用

Web 主站、Android、iOS 不要各自定义一套提交逻辑。  
所有客户端统一走同一套 `/api/v1` 接口，工作流列表、参数 schema、任务提交、任务状态、历史任务、资源删除都必须一致。

### 3.2 参数只暴露“可见 / 激活”项

客户端只看到后台参数配置里允许暴露的字段：

- `visible = true`
- `active = true`

未暴露参数不应出现在客户端表单中。

### 3.3 工作流是授权对象

工作流不只是一个模板，它也是权限对象。  
每个工作流可以单独授权给：

- 某个用户
- 某个用户等级
- 某个用户组

### 3.4 任务闭环必须完整

从提交到完成，客户端必须能看到：

- 排队状态
- 执行状态
- 失败原因
- 结果文件
- 历史记录
- 删除操作

---

## 4. 推荐的系统分层

### 4.1 API 层

统一提供：

- 登录 / 注册 / 刷新 token
- 工作流列表、参数、授权信息
- 提交任务
- 查询任务状态
- 查询历史任务
- 删除任务
- 资源管理
- 点数查询

### 4.2 业务层

负责：

- 参数过滤
- 默认值补全
- 随机值生成
- 文件上传映射
- 工作流注入
- 队列调度

### 4.3 执行层

负责：

- 将任务转成 ComfyUI 可执行 payload
- 处理子图、禁用节点、节点映射
- 提交到目标 GPU 节点

### 4.4 权限层

负责：

- 用户登录态
- 角色
- 等级
- 组别
- 工作流可见范围
- 点数/额度限制

---

## 5. 建议的权限模型

### 5.1 用户维度

支持按用户单独授权：

- 某个用户可看某个工作流
- 某个用户只能用部分工作流
- 某个用户有更高点数额度

### 5.2 等级维度

支持按用户级别授权：

- 普通用户
- 会员
- 高级用户
- 内测用户

### 5.3 组别维度

支持按组授权：

- 设计组
- 运营组
- 内测组
- VIP 组

### 5.4 工作流授权建议

每个工作流建议维护这些字段：

- 是否启用
- 可见范围
- 可提交范围
- 单次消耗点数
- 排队优先级

---

## 6. 客户端规划

### 6.1 Web 主站

Web 端建议承担完整功能：

- 登录 / 注册
- 工作流列表
- 参数填写
- 图片 / 视频上传
- 任务提交
- 历史任务
- 结果查看
- 自己生成的资源删除
- 点数展示
- 排队进度展示

### 6.2 Android App

Android 端建议先做“轻量主流程”：

- 登录 / 注册
- 浏览可用工作流
- 提交任务
- 查看任务队列和进度
- 查看历史结果
- 删除资源
- 查看点数

### 6.3 iOS App

iOS 端与 Android 使用同一份 API 契约，尽量保持一致的：

- 登录态
- 工作流展示
- 参数表单
- 提交逻辑
- 历史任务逻辑

---

## 7. 管理端规划

管理端建议重点做这几块：

### 7.1 用户管理

- 注册用户审核
- 用户启用 / 禁用
- 用户点数增减
- 用户等级调整
- 用户组管理

### 7.2 工作流管理

- 工作流导入
- 工作流参数配置
- 可见 / 激活控制
- 工作流授权到用户 / 等级 / 组别
- 依赖检测
- 缺失模型下载

### 7.3 任务管理

- 所有用户任务
- 排队状态
- 失败原因
- 任务删除
- 批量操作

### 7.4 资源管理

- 用户生成资源
- 任务结果文件
- 删除资源
- 目录 / 模型管理

---

## 8. 推荐的开发顺序

### 第一阶段：API 契约稳定

目标：

- 确认所有客户端都用同一套接口
- 确认工作流参数 key 的协议
- 确认任务状态、历史任务、删除资源、点数接口

交付物：

- API 接口说明
- 工作流调用模板
- 参数契约说明

### 第二阶段：权限与授权

目标：

- 按用户 / 等级 / 组别控制工作流可见性
- 按工作流控制可用范围

交付物：

- 工作流授权表
- 用户等级规则
- 工作流开放清单

### 第三阶段：Web 主站

目标：

- 完整跑通用户主流程

交付物：

- 登录 / 注册
- 工作流卡片页
- 任务提交页
- 任务队列页
- 历史任务页
- 资源页

### 第四阶段：Android / iOS

目标：

- 复用同一套 API 契约
- 只做客户端体验层

交付物：

- 登录
- 工作流提交
- 任务状态
- 历史记录
- 资源删除

### 第五阶段：运营与优化

目标：

- 让系统可长期运营

交付物：

- 审计日志
- 点数流水
- 队列监控
- 失败任务分析
- 模型依赖健康检查

---

## 9. 目前建议优先做的几件事

1. 把当前启用工作流的 API 调用说明做成唯一主文档。
2. 定义工作流授权模型，支持用户 / 等级 / 组别。
3. 固化任务状态和结果文件的统一接口。
4. 给 Android / iOS 输出同一套请求体示例。
5. 管理端补工作流开放范围和用户点数管理。

---

## 10. 结论

这个项目下一阶段的关键，不是“再多做几个页面”，而是：

- 统一 API 契约
- 统一工作流权限
- 统一任务闭环
- 统一三端接入方式

这样才能从“管理型站点”升级为“可直接给用户使用的多端生成平台”。
