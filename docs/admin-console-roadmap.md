# 管理端改进路线图

> 目标：把当前可用的管理能力继续补完整，让 Web 管理端真正支撑多端产品的运营、授权和排队管理。

---

## 1. 当前管理端的核心定位

管理端不只是“工作流编辑页”，它应该是整个系统的运营控制台，负责：

- 工作流开放范围
- 用户权限与等级
- 点数与额度
- 任务排队与失败处理
- 资源与模型管理
- 统计与审计

---

## 2. 优先级排序

### P0：工作流授权闭环

这是后续 Android / iOS 动态渲染的前提。

需要支持：

- 工作流对哪些用户可见
- 工作流对哪些用户可提交
- 工作流对哪些用户等级可用
- 工作流对哪些用户组可用
- 工作流是否启用
- 工作流是否对外展示
- 单次消耗点数
- 排队优先级

建议页面：

- 工作流列表
- 工作流详情
- 工作流授权面板

---

### P0：用户管理

需要支持：

- 用户注册审核 / 激活
- 用户禁用 / 解禁
- 用户等级设置
- 用户组设置
- 点数充值 / 扣减 / 赠送
- 剩余点数查看
- 用户可用工作流查看

建议页面：

- 用户列表
- 用户详情
- 用户等级与组别
- 点数流水

---

### P1：工作流参数管理

当前已经有基础，但还可以继续优化。

建议保留的参数项：

- 可见
- 激活
- 必填
- 默认值
- 随机数

建议展示方式：

- 主节点默认展开
- 子节点默认折叠
- 灰色节点保留但弱化显示
- 只对壳子节点提供展开子节点的入口

建议页面：

- 参数配置页
- 参数测试页
- 工作流预览页

---

### P1：任务监控与操作

需要支持：

- 所有任务列表
- 排队中 / 执行中 / 成功 / 失败
- 当前节点
- 进度
- 失败原因
- 停止任务
- 重新提交
- 删除任务
- 批量删除

建议页面：

- 任务列表
- 任务详情
- 失败任务筛选

---

### P1：资源管理

需要支持：

- 用户生成的图片 / 视频 / 音频
- 任务结果文件
- 删除自己生成的资源
- 管理员统一清理
- 存储占用查看

建议页面：

- 资源列表
- 存储空间占用
- 文件清理入口

---

### P2：模型与依赖

需要支持：

- 模型依赖检测
- 缺失模型下载
- 模型使用关系
- 模型目录管理

建议页面：

- 模型列表
- 依赖缺失列表
- 下载任务监控

---

### P2：统计与审计

需要支持：

- 今日任务数
- 成功率 / 失败率
- 点数消耗
- 工作流使用量
- 用户活跃度
- 操作日志

建议页面：

- 仪表盘
- 统计图表
- 操作日志

## 4. 工作流授权建议

建议每个工作流保存这些授权字段：

- `enabled`
- `visible`
- `canSubmit`
- `creditCost`
- `priority`
- `allowedUserIds`
- `allowedLevelIds`
- `allowedGroupIds`

这样客户端和管理端都能稳定判断：

- 看不看得见
- 能不能提交
- 花多少点数
- 优先级多高

---

## 5. 与 Android / iOS 的关系

管理端的职责不是“把页面做好看”，而是：

- 输出工作流可见范围
- 输出参数 schema
- 输出权限规则
- 输出点数规则

客户端只负责：

- 登录
- 拉工作流列表
- 拉参数 schema
- 动态渲染表单
- 提交任务
- 查看队列和结果

---

## 6. 近期建议

### 先做

- 工作流授权
- 用户等级 / 组别
- 点数管理
- 任务监控

### 再做

- 资源管理
- 模型依赖与下载
- 统计与审计

### 最后做

- 更精细的运营面板
- 自动策略
- 使用趋势分析

---

## 7. 结论

管理端下一步最该补的，不是再做新的“功能点”，而是把：

- 工作流授权
- 用户权限
- 点数额度
- 任务闭环

这四条线收完整。

只要这四条线稳了，Web、Android、iOS 就都能围着同一套 API 和权限体系跑起来。
