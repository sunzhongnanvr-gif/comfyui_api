# 开发计划

> 目标：在当前已可用的中转站基础上，继续把“工作流授权、任务闭环、资源管理、运营控制”做完整。
>
> 当前范围：**中转站后台 + 管理端**
>
> 暂不纳入：Web 主站体验改版、Android App、iOS App 的界面开发
>
> 但 **主站和 App 的 API 契约必须统一**，后续所有客户端都只吃同一套接口。

---

## 1. 项目现状

目前系统已经具备：

- 工作流导入、解析、参数配置
- 任务提交、执行、排队、状态查询
- 模型依赖检测与下载
- 用户登录 / 注册
- 点数 / 积分基础能力
- 管理端基础页面

接下来要补的，不是零散功能，而是把它们收成一套稳定的后台能力：

- 工作流授权
- 工作流 manifest
- 用户权限 / 等级 / 组别
- 任务闭环
- 资源闭环
- 模型与依赖运营

---

## 2. 总体原则

### 2.1 后端驱动

所有工作流能力、参数 schema、授权范围都由服务端输出，前端不硬编码节点细节。

### 2.2 权限先行

工作流先决定“谁能看、谁能提交”，再决定“显示哪些参数”。

### 2.3 前后端契约统一

主站 Web、Android、iOS 统一使用同一套 API：

- 同一个登录 / 注册 / 刷新 token
- 同一个工作流列表接口
- 同一个工作流参数清单接口
- 同一个任务提交接口
- 同一个任务状态 / 历史 / 删除接口

客户端可以长得不一样，但请求协议不能不一样。

### 2.4 任务闭环优先

提交任务只是开始，必须能看到：

- 排队
- 执行
- 失败
- 结果
- 删除
- 资源归档

### 2.5 运营可控

管理端要能管：

- 用户
- 工作流
- 点数
- 模型
- 任务
- 资源

---

## 3. 第一阶段：工作流授权与 manifest

### 目标

让系统能够按用户、等级、组别返回不同工作流，并且每个工作流都能返回可渲染的参数清单。

### 需要完成

- 定义工作流 manifest 结构
- 提供工作流列表接口
- 提供工作流参数清单接口
- 支持按用户 / 等级 / 组别过滤工作流
- 支持工作流是否启用、是否可见、是否可提交

### 建议交付

- `GET /api/v1/workflows`
- `GET /api/v1/workflows/{slug}`
- `GET /api/v1/workflows/{slug}/inputs`
- 管理端工作流授权页

### 结果

客户端不再硬编码工作流，所有工作流能力都从服务端动态获取。

---

## 4. 第二阶段：用户、等级、组别、点数

### 目标

把用户管理和点数体系补成可运营状态。

### 需要完成

- 用户注册审核 / 激活
- 用户禁用 / 解禁
- 用户等级管理
- 用户组管理
- 点数充值 / 扣减 / 赠送
- 点数流水记录
- 用户剩余点数可查

### 建议交付

- 用户列表页
- 用户详情页
- 等级管理页
- 组别管理页
- 点数流水页

### 结果

工作流可以按用户状态和额度精细控制，后续三端接入时也能直接复用。

---

## 5. 第三阶段：任务队列与任务闭环

### 目标

把任务从“提交后生成一条记录”升级为完整可运营的任务系统。

### 需要完成

- 排队状态
- 当前执行节点
- 进度显示
- 失败原因
- 停止任务
- 重新提交
- 删除任务
- 批量删除

### 建议交付

- 任务列表页
- 任务详情页
- 失败任务筛选页
- 队列监控页

### 结果

管理端能随时知道任务在哪个节点、为什么失败、哪些可以重试、哪些可以清理。

---

## 6. 第四阶段：资源管理

### 目标

把生成结果、上传文件、临时文件、历史结果纳入统一资源管理。

### 需要完成

- 图片 / 视频 / 音频资源列表
- 自己生成资源删除
- 管理员统一清理
- 结果文件归档
- 存储占用查看

### 建议交付

- 资源列表页
- 资源预览页
- 存储占用页
- 清理操作页

### 结果

系统不会只靠任务表活着，而是真正形成“任务 -> 结果 -> 资源”的闭环。

---

## 7. 第五阶段：模型与依赖运营

### 目标

把模型依赖检测、下载、目录管理做成运营能力，而不是临时工具。

### 需要完成

- 模型依赖检查
- 缺失模型预览
- 缺失模型下载
- 模型使用关系
- 模型目录管理
- 下载任务监控

### 建议交付

- 模型列表页
- 依赖缺失页
- 下载任务页
- 模型使用分析页

### 结果

工作流导入后，系统能持续判断“缺什么、在哪、怎么补”，支撑多 GPU 节点调度。

---

## 8. 第六阶段：管理端工作流编辑体验

### 目标

让管理端像 ComfyUI 一样可理解，但又不暴露过多实现细节。

### 需要完成

- 主节点默认展开
- 子节点默认折叠
- 灰色禁用节点弱化显示
- 壳子节点可展开子节点模式
- 参数可见 / 激活 / 必填 / 随机数
- 参数默认值与类型统一展示

### 建议交付

- 工作流参数配置页
- 工作流预览页
- 工作流测试页

### 结果

管理员看到的是“可运营的工作流”，不是一堆难以理解的节点原文。

---

## 9. 推荐实施顺序

建议按这个顺序推进：

1. 工作流授权与 manifest
2. 用户 / 等级 / 组别 / 点数
3. 任务队列与任务闭环
4. 资源管理
5. 模型与依赖运营
6. 管理端工作流编辑体验

这个顺序的好处是：

- 先把数据契约定住
- 再把运营规则定住
- 最后才优化界面

---

## 10. 当前不纳入本阶段的内容

以下内容先不放进这版开发计划：

- Web 主站用户体验改版
- Android App 客户端
- iOS App 客户端

这些会放到后续单独计划里，在后端契约和管理端稳定后再推进。

---

## 11. 这版计划的判断标准

如果下面这些能稳定做到，就说明中转站和管理端已经进入“可运营”阶段：

- 任意用户只能看到自己能用的工作流
- 参数只暴露可见 / 激活项
- 任务能排队、能看进度、能看失败原因
- 结果能查、能删、能归档
- 点数、等级、组别可控
- 缺失模型可自动检测并补齐

---

## 12. 结论

下一阶段的重点不是继续堆功能，而是把后台能力收紧成一套稳定的运营系统：

- 工作流能管
- 用户能管
- 任务能管
- 资源能管
- 模型能管

这五条线稳了，后面的多端扩展就顺很多。
