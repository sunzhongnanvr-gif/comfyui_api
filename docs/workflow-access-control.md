# 工作流授权模型

> 这份是后台参考规范，核心内容已并入 [Android 调用说明](./android-workflow-api.md)。  
> 如果你是给客户端接入，请优先看 Android 主文档，不要再把这份当成第二套入口。

> 目标：让不同用户、不同等级、不同组别看到不同的工作流，并且能控制可提交范围、点数和优先级。

---

## 1. 核心原则

工作流不是静态模板，而是一个可授权对象。  
同一个工作流对不同用户可以有不同开放范围。

---

## 2. 授权维度

### 2.1 单用户授权

适合：

- 内测
- 单独客户
- 定向开放工作流

规则：

- 某个用户是否可见
- 某个用户是否可提交
- 某个用户是否能看到结果资源

### 2.2 用户等级授权

适合：

- 普通用户
- 会员
- 高级用户
- 内测用户

规则：

- 等级低的看不到高级工作流
- 等级高的可看到更多工作流
- 高级工作流可以额外设置更高点数

### 2.3 用户组授权

适合：

- 设计组
- 运营组
- 内测组
- VIP 组

规则：

- 组内共享可见工作流
- 组外默认不可见

---

## 3. 工作流建议字段

建议每个工作流都维护这些授权字段：

```ts
enabled: boolean
visible: boolean
canSubmit: boolean
creditCost: number
priority: number
allowedUserIds: string[]
allowedLevelIds: string[]
allowedGroupIds: string[]
```

---

## 4. 过滤规则建议

服务端返回 `GET /api/v1/workflows` 时，建议按以下优先级过滤：

1. `enabled = true`
2. 用户属于允许的单用户范围，直接放行
3. 用户等级命中允许范围，放行
4. 用户组命中允许范围，放行
5. 不命中则隐藏

---

## 5. 提交规则建议

即使工作流在列表里可见，也不代表可提交。  
建议再单独校验：

- `canSubmit = true`
- 用户状态正常
- 用户积分足够
- 工作流当前启用
- 工作流参数校验通过

---

## 6. 客户端表现建议

客户端可以把工作流分成三种状态展示：

- 可见可提交
- 可见不可提交
- 不可见

这样用户体验会更清楚，管理端也更容易排查权限问题。
