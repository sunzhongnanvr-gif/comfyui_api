## 三、安全审查 🔴 (严重)

### 3.1 关键安全问题

#### 🔴 CRITICAL-1: `.env` 文件被提交到仓库

```
.env 存在于项目根目录，包含:
  - JWT_SECRET=your-super-secret-jwt-key-change-me-in-production
  - ADMIN_PASSWORD=admin123
  - DATABASE_URL=postgresql://postgres:postgres123@...
```

**影响**: 所有仓库协作者/历史提交者可直接访问生产数据库和管理员账户。  
**修复**: 立即加入 `.gitignore` + 轮换所有凭证。

#### 🔴 CRITICAL-2: 缺少 `.gitignore`

**影响**: `.env`、`node_modules/`、`dist/`、`postgres/data/`、`volumes/results/`、`.sandbox-home/` 等敏感/冗余文件可能被提交。  
**修复**: 创建 `.gitignore` 并清理 git 历史。

#### 🔴 CRITICAL-3: JWT 实现有严重漏洞

```typescript
// src/middleware/auth.ts
const token = authHeader.startsWith('Bearer ') ? authHeader.slice(7) : authHeader;
const decoded = jwt.verify(token, process.env.JWT_SECRET || 'fallback-secret'); // ⚠️ fallback
req.user = decoded as { userId: string };
```

**问题**:
1. **Hardcoded fallback secret** — 当 `JWT_SECRET` 未设置时使用 `'fallback-secret'`，攻击者可伪造任意 Token
2. **未验证 Token 类型** — access token 和 refresh token 用同一密钥验证，无法区分用途
3. **未检查 Token 黑名单** — 登出/密码修改后旧 Token 仍然有效
4. **密码重置 Token 复用 JWT Secret** — passwordResetToken 和 access token 共享同一签名密钥

#### 🟠 HIGH-4: 管理员初始化无保护

```typescript
// src/config/seed.ts
if (!adminUser) {
  const hashedPassword = await bcrypt.hash(password, 10);
  await prisma.user.create({ ... });
}
```

每次进程启动都会用环境变量中的密码覆盖 admin 账户密码，无审计、无确认。

#### 🟠 HIGH-5: 登录暴力破解防护薄弱

```typescript
// routes/auth.routes.ts 第 47-58 行
loginMaxRetries = 5          // 硬编码上限
loginCooldownSeconds = 900   // 无 IP/账号维度的窗口计数
```

- 每次登录成功/失败都重置计数器
- 无渐进式延迟策略
- 无账户锁定机制
- **`recordFailedLogin` 函数为空实现** (第 38-40 行: `// TODO: more sophisticated tracking`)

#### 🟠 HIGH-6: 文件上传无扩展名校验

```typescript
// src/middleware/upload.ts
const uploadLocal = multer({
  dest: path.join(storagePath, 'temp-uploads'),
  limits: { fileSize: MAX_FILE_SIZE * 1024 * 1024 },
  // ⚠️ 无 fileFilter 校验扩展名/MIME
});
```

攻击者可上传恶意脚本（`.js`, `.sh`, `.php`）并通过 `file.routes.ts` 的 `serveUpload` 端点执行。

#### 🟠 HIGH-7: Admin 端面无路由守卫

```typescript
// src/routes/admin.routes.ts
// 任何携带有效 JWT 的用户都可访问所有 admin API
// 未检查 user.role === 'admin'
```

**所有 admin 端点（包括用户删除、系统配置、缓存清理）对普通用户开放。**

#### 🟡 MEDIUM-8: CORS 过于宽松

```typescript
app.use(cors({ origin: '*', methods: ['GET', 'POST', 'PUT', 'DELETE'], credentials: true }));
```

`origin: '*'` + `credentials: true` 是无效组合（浏览器会拒绝），但若前端 SPA 直接暴露，任何第三方页面都能调用 API。

#### 🟡 MEDIUM-9: HSTS/安全 Header 缺失

未设置 `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`, `Strict-Transport-Security`。

#### 🟡 MEDIUM-10: 密码哈希策略

```typescript
// src/routes/auth.routes.ts
const hashedPassword = await bcrypt.hash(password, 10);  // OK
// src/config/seed.ts
const hashedPassword = await bcrypt.hash(password, 10);  // OK
// web/src/lib/auth.ts
const hashedPassword = await bcrypt.hash(password, 10);  // OK
```

bcrypt rounds=10 尚可接受，但三处独立实现增加维护风险。

## 四、代码质量审查

### 4.1 关键代码问题

#### 问题 1: Routes 文件过长，违反单一职责

```

admin.routes.ts  : 479 行 (14 个端点)

```

Controller 逻辑全部写在 route handler 中，不可测试。

#### 问题 2: TaskExecutor 缺少关键保护

```typescript
// src/services/task-executor.ts
public async execute(workflowId: string, ...): Promise<TaskResult> {
  // 1. 提交到 ComfyUI
  // 2. 轮询进度
  // 3. 保存结果
  // ⚠️ 无超时中断后清理 ComfyUI 侧的排队任务
  // ⚠️ prompt_id 未存入 Task 记录，无法反向追踪
  // ⚠️ postProcess 未检查 ComfyUI 返回的消息错误
}
```

#### 问题 3: Workflow 双解析器共存

```
src/utils/workflow-parser.ts    // 旧版 (685 行)
src/utils/workflow-parser-v2.ts // 新版 (717 行)
```

两个解析器功能重复，`v2` 号称支持"高级嵌套"但旧版未标记废弃。增加维护负担和潜在不一致。

#### 问题 4: 队列管理脆弱

```typescript
// src/services/task-executor.ts: QueueManager
private queue: TaskQueueItem[] = [];  // ⚠️ 纯内存队列
public drain(): TaskQueueItem | null  // ⚠️ 无并发锁
```

- 多实例部署时队列不同步
- 进程重启队列丢失
- `drain()` 无互斥锁，理论上可能并发取出同一项

#### 问题 5: 错误处理不一致

```typescript
// 有些路由用 try-catch + 手动 HTTP error
// 有些依赖全局 errorHandler 中间件
// 有些直接 throw new Error() (会被 Prisma Client 包装成不同格式)
// 前端 error-handler.ts 有响应校验但过于脆弱
```

#### 问题 6: TypeScript 类型安全

- `any` 类型散布：`workflow-parser-v2.ts` 中大量 `as any`
- `workflowData` 字段类型仅为 `any`，未定义 Zod schema
- ComfyUI API 响应未做强类型约束

#### 问题 7: 资源泄漏风险

```typescript
// src/utils/object-info-cache.ts
private cache: Map<string, any> = new Map();  // 无限增长
private cacheTimeout: NodeJS.Timeout | null = null; // 60s TTL
```

Map 只在写入时重置定时器，高频写入场景下旧值永不回收。
